# TEE-hosted private agent architecture

## One-line claim

The same `pacto-bot-api` daemon, NIP-46 bunker, and bot handler that run on a laptop or VPS today can run unchanged inside a Trusted Execution Environment (TEE). The TEE adds **isolation** (host OS and cloud operator cannot read memory), **sealed storage** (MLS keys are encrypted at rest), and **attestation** (users can verify the exact code that is running). No re-architecture of the daemon or the governance bot is required.

## Why a TEE?

The daemon already keeps `nsec`, bunker URIs, and the HTTP token out of logs and handler responses. For MLS group messaging, however, the daemon also holds a new class of secrets that cannot be delegated to a remote NIP-46 bunker:

- Ed25519 credential and signature keys
- X25519 leaf encryption keys
- Per-epoch group secrets derived during Welcome processing
- The persisted group state in `$DATA_DIR/<bot_id>/vector-mls.db`

These are local-only keys generated by OpenMLS inside the `mdk-core` engine. If the host is compromised, an attacker who can read `vector-mls.db` or the daemon's memory can decrypt already-posted and future channel messages until the group is externally re-keyed. The existing `0o600` file permission and `zeroize` memory hygiene are necessary but not sufficient against a hostile host or cloud administrator.

A TEE moves the trust boundary from the operating system to the hardware. The daemon's memory and the database it holds are encrypted and isolated from the host, and users can check that the running enclave matches the published daemon binary.

## What the TEE protects

| Asset | Threat without TEE | TEE protection |
|---|---|---|
| `vector-mls.db` on disk | Host/cloud operator reads the file | Sealed at rest; only the TEE can unwrap the encryption key |
| MLS keys in memory | Memory dump, debugger, swap leak | Memory is encrypted and inaccessible to the host OS |
| Nostr `nsec` (if used) | Same as above | Same as above (still recommended: use `bunker_remote` in production) |
| Handler config / RPC URLs | Host operator reads env or file | Inside the TEE boundary; secrets never cross to the parent |
| Execution integrity | Host replaces daemon with a malicious binary | Attestation proves the exact image is running |

## What does not change

The TEE is a deployment target, not a code redesign. The following stay identical:

- `pacto-bot-api` source code, JSON-RPC contract, and `schemas/jsonrpc.json`
- `pacto-bot-admin` lifecycle commands
- The `mdk-core` / `MlsEngineHandle` / `spawn_blocking` engine pattern
- The NIP-46 bunker (local or remote) and the existing signer backends
- The `crates/governance-bot` handler and its cadence/formatter logic
- Data directory layout: `$DATA_DIR/agent.db`, `$DATA_DIR/<bot_id>/vector-mls.db`, etc.
- File permissions (`0o600` for secrets, `0o700` for per-bot directories)

The only new components are the TEE packaging, a sealed-storage unlock step at boot, and an attestation endpoint that answers the `!verify` challenge.

## Platform comparison

| Platform | Isolation model | Sealed storage | Attestation | Memory / CPU constraints | Packaging complexity |
|---|---|---|---|---|---|
| **AWS Nitro Enclaves** | Separate, hardened VM with no direct I/O; only a `vsock` channel to the parent EC2 instance | AWS KMS encrypt-to-enclave via attestation; or encrypted parent-attached volume + key unwrapped with `nsm_get_attestation_doc()` | Nitro hypervisor signs an attestation document containing PCR measurements; verified by AWS KMS or the user's client | Up to full instance resources, but enclave memory and CPU are carved out from the parent instance | Medium: build an Enclave Image File (EIF), run a parent proxy, manage vsock |
| **Azure Confidential VMs (SEV-SNP)** | Full VM with memory encrypted from the hypervisor | Confidential OS disk + vTPM-sealed LUKS/dm-crypt; or Azure Key Vault with attestation policy | Microsoft Azure Attestation (MAA) returns a token signed by Azure; local vTPM quote also available | Full VM size (e.g., 4+ vCPU, 16+ GB) | Low: mostly a normal VM with a confidential SKU and disk type |
| **Intel SGX / TDX** | Process-level enclave inside a host application | `sgx_seal_data` to MRENCLAVE/MRSIGNER; data stored encrypted on host | Intel IAS or DCAP quote signed by the quoting enclave | Historically 128–256 MB EPC; TDX lifts this but tooling is newer and scarcer | High: requires an enclave loader (Gramine, SCONE, Open Enclave) and careful EPC management |

**Recommendation:** Use **AWS Nitro Enclaves** as the primary target. It provides a clean isolation model (no direct network or storage), a strong attestation document signed by the Nitro hypervisor, and an AWS KMS integration that makes sealed-storage key unwrapping straightforward. It is the best fit for a daemon that already relies on network sockets and a local SQLite file.

**Azure Confidential VMs** are the best secondary option: they are the easiest to deploy (a normal VM with confidential disk) and scale to the full memory needs of the daemon, but their attestation is at VM granularity and their sealing is tied to Azure-specific vTPM/Key Vault policies.

**Intel SGX** is not recommended for this workload because the daemon + OpenMLS + SQLite + nostr-sdk runtime comfortably exceeds the classic EPC memory envelope, and the packaging complexity (loader, EDG, memory management) is not justified for the extra isolation over Nitro Enclaves or Azure CVMs.

## Packaging the stack

All three components are packaged as ordinary Linux binaries. No code changes are required.

### General layout (all platforms)

```
/pacto-stack/
  ├── bin/
  │   ├── pacto-bot-api      # daemon
  │   ├── pacto-bot-admin    # CLI (used at setup/bake time, optional at runtime)
  │   └── governance-bot     # example handler (or your handler binary)
  ├── config/
  │   └── pacto-bot-api.toml # signed/encrypted or baked into the image
  └── data/
      ├── agent.db
      └── <bot_id>/
          └── vector-mls.db  # sealed at rest
```

### AWS Nitro Enclaves packaging

1. Build a Docker image containing the three binaries and a startup script.
2. Convert the image to an **Enclave Image File (EIF)** with `nitro-enclaves-cli build-enclave`.
3. The EIF hash becomes the published **measurement** (PCR0) that users compare against.
4. Run the enclave from the parent EC2 instance. The enclave has no network or block devices; the parent proxies network traffic via `vsock` (e.g., `socat` or the Nitro Enclaves proxy) and stores the encrypted `data/` directory.

### Azure CVM packaging

1. Build a standard VM image (e.g., Ubuntu with the binaries installed).
2. Deploy with a **Confidential VM** SKU and a **confidential OS disk**.
3. Enable vTPM and secure boot. The vTPM seals the disk encryption key.
4. The bot runs as a normal systemd service; no vsock proxy is needed because the VM has full networking.

### SGX packaging (not recommended)

1. Use Gramine or Open Enclave to load the daemon binary inside an enclave.
2. Seal the `vector-mls.db` encryption key with `sgx_seal_data` and store the encrypted blob on the host.
3. Provide an attestation quote endpoint over the host-facing socket.

## Sealed storage for MLS state

The daemon's `$DATA_DIR` contains two secrets classes:

- **Configuration signing material:** `pacto-bot-api.toml` (should use a bunker backend, not `nsec`), and the HTTP token file.
- **MLS state:** `$DATA_DIR/<bot_id>/vector-mls.db` and the in-process OpenMLS keys derived from it.

Sealed storage protects the MLS state at rest while keeping the same runtime behavior.

### Recommended sealed-storage pattern

1. At **image build time**, the daemon binary and config are measured (Nitro PCRs, Azure secure boot PCRs, or SGX MRENCLAVE).
2. At **runtime boot**, a small wrapper inside the TEE calls the platform attestation API and receives an attestation document/quote.
3. The wrapper uses the attestation document to authenticate to a key service (AWS KMS, Azure Key Vault, or an internal HSM) and unwraps a **data encryption key (DEK)**.
4. The DEK decrypts the `data/` volume or directory (e.g., LUKS/dm-crypt, or an application-level AES-GCM wrapper around `vector-mls.db`).
5. The daemon starts and reads/writes `vector-mls.db` as it does today. The `0o600` permission is still enforced; the TEE adds the at-rest encryption layer.
6. Before shutdown, the daemon flushes SQLite WAL and closes the database. The DEK is dropped from memory.

### Platform-specific sealing notes

| Platform | Where the DEK lives | How the TEE unwraps it |
|---|---|---|
| AWS Nitro Enclaves | AWS KMS as a `CiphertextBlob` with a key policy requiring a specific PCR0 value | `nsm_get_attestation_doc()` → KMS `Decrypt` or `GenerateDataKey` with attestation condition |
| Azure CVM | Azure Key Vault or vTPM-backed key storage | MAA attestation token → Key Vault release policy; or vTPM unseal |
| Intel SGX | Host disk as an encrypted sealed blob | `sgx_unseal_data()` bound to MRENCLAVE/MRSIGNER |

The important invariant is that **the unwrapped key exists only in TEE memory**. The persistent `data/` directory can live on ordinary cloud storage because it is encrypted with the DEK.

## Attestation and the `!verify` concept

Before a squad invites a TEE bot, members need proof that the bot is running the expected code inside a real TEE. The `!verify` command is a challenge-response protocol:

1. **User client sends a challenge.** It generates a random nonce (e.g., 32 bytes) and sends it to the bot via a DM or a dedicated `/verify` endpoint.
2. **Enclave produces an attestation document.** The daemon calls the platform attestation API (`nsm_get_attestation_doc` for Nitro, MAA for Azure, `sgx_get_quote` for SGX) with the nonce embedded in the request.
3. **Platform root of trust signs the report.** The report contains the nonce and the measurement hash of the running image (e.g., Nitro PCR0, Azure launch measurement, or SGX MRENCLAVE).
4. **User client verifies the report.** It checks:
   - The signature against the platform's root of trust (AWS Nitro public key, Azure MAA, Intel IAS/DCAP).
   - The nonce matches the one it sent (freshness / no replay).
   - The measurement hash equals the published value for the release (e.g., the EIF hash in the release notes).
5. **Client accepts or rejects.** If the measurement matches, the user knows the exact daemon binary is running inside a genuine TEE. If not, the bot is treated as untrusted.

The measurement hash is published with each release (e.g., in `docs/pacto-bot-admin-llms.txt`, release notes, or a signed attestation metadata file). It should be reproducible from the same source tree and build configuration.

## Reference deployment: AWS Nitro Enclaves

This is the recommended production-shaped deployment.

### Build

```bash
docker build -t pacto-tee:latest -f tee/Dockerfile .
nitro-enclaves-cli build-enclave \
  --docker-uri pacto-tee:latest \
  --output-file pacto-tee.eif
# Record PCR0 for the release notes.
nitro-enclaves-cli describe-enclaves
```

The `Dockerfile` installs the three binaries, sets `/pacto-stack/data/` permissions to `0o700`, and copies a startup script that:

1. Calls `nsm_get_attestation_doc()`.
2. Unwraps the DEK from AWS KMS using the PCR0-bound policy.
3. Opens the LUKS/dm-crypt volume on the parent-attached encrypted volume.
4. Starts `pacto-bot-api` and `governance-bot` (or leaves the handler to the operator).

### Parent EC2 instance

- Run the parent on a Nitro Enclaves-enabled instance type (e.g., `m5.xlarge` or larger).
- Store the encrypted `data/` volume on an EBS volume or EFS attached to the parent.
- Run a `vsock` proxy on the parent to allow the enclave to reach:
  - Nostr relays over WSS
  - The remote NIP-46 bunker over WSS
  - The EVM RPC endpoint over HTTPS
- The parent never sees plaintext keys or `vector-mls.db`.

### Enclave launch

```bash
nitro-enclaves-cli run-enclave \
  --eif-path pacto-tee.eif \
  --cpu-count 2 \
  --memory 4096 \
  --enclave-cid 16
```

### Runtime flow

```
Parent EC2                          Enclave
   |                                    |
   |--- vsock proxy (WSS/HTTPS) ------->|--- pacto-bot-api connects to relays/bunker/RPC
   |                                    |--- daemon + handler run normally
   |                                    |--- vector-mls.db lives on encrypted volume
   |<--- encrypted data volume I/O ---|    (keys only in TEE memory)
```

### First-time setup

1. Build the EIF and record PCR0.
2. Create an AWS KMS key with a key policy that permits `Decrypt` only when the Nitro attestation document contains the expected PCR0.
3. Create the encrypted data volume: boot a temporary enclave with the same PCR0, generate a DEK via KMS, format the volume, and shut it down.
4. Launch the production enclave. The KMS policy ensures only an enclave with the correct PCR0 can unwrap the DEK.
5. Use `pacto-bot-admin` from inside the enclave (or from a pre-baked config) to create the bot identity and publish the KeyPackage.

## Azure CVM alternative

For teams that prefer a simpler path or are already in Azure:

1. Deploy an Azure Confidential VM (e.g., `Standard_DC2as_v5`) with a confidential OS disk and vTPM enabled.
2. Install the daemon and handler as a normal systemd service.
3. Configure full-disk encryption (Azure Disk Encryption or confidential disk encryption) so the OS disk is sealed to the vTPM.
4. For `!verify`, expose an attestation endpoint that calls the Azure Attestation Service (MAA) with a nonce and returns the attestation token + launch measurement.
5. The bot has full network access, so no parent proxy is needed.

Trade-off: the attestation boundary is the whole VM rather than a single purpose-built enclave, and the sealing chain is tied to Azure-specific vTPM/Key Vault policies. It is still a strong, production-ready option.

## Operational notes

### Backups and recovery

- Back up the encrypted `data/` volume only. The DEK is not backed up; it is re-derived from the attestation at each boot.
- If `vector-mls.db` becomes corrupted, delete the file and re-invite the bot. The TEE does not change this recovery step; it only protects the file at rest.
- If you lose the KMS key policy, the sealed data is unrecoverable. Document the key ARN and policy in a separate, access-controlled location.

### Re-keying the MLS group

- The TEE protects the key file, but it does not change the MLS group re-key semantics. If the bot is decommissioned or suspected compromised, remove it from the Squad and re-invite a fresh identity with a new `vector-mls.db`.
- The bot's Nostr signing key (if using a bunker) remains at the bunker; only the MLS state is re-created.

### Monitoring

- Emit health metrics from the parent as usual. The daemon's `pacto-bot-admin diagnose --format json` output remains safe to share because it contains no secrets.
- Log attestation failures and unexpected PCR changes separately; they are high-signal security events.

### Cost and sizing

- Nitro Enclaves carve CPU and memory out of the parent instance. Size the parent so the enclave has enough memory for the daemon + SQLite WAL + OpenMLS state (start with 4 GB, scale up for many bots).
- Azure CVMs require a confidential SKU; plan for the same VM size as a normal VPS.

## Follow-up sprint scope

This brief is a design document. A follow-up sprint to make it runnable would produce:

1. **Packaging automation:** `tee/Dockerfile`, `build-eif.sh`, and GitHub Actions steps to build the EIF and publish its PCR0 hash.
2. **Sealed-storage wrapper:** a small Rust binary or shell script that runs before the daemon to perform the attestation → DEK unwrap → volume mount dance.
3. **Attestation endpoint:** a tiny HTTP server inside the daemon (or a sidecar) that answers the `!verify` challenge with a signed attestation document.
4. **KMS/key-vault policy templates:** Terraform or CloudFormation for AWS KMS key policies and Azure MAA policies.
5. **Deployment guides:** step-by-step instructions for AWS Nitro Enclaves and Azure CVMs, including parent proxy setup, networking, and handler connection.
6. **Integration tests:** mock attestation documents and a test that verifies the `!verify` nonce and measurement flow.

None of these require changing the daemon's core logic, the JSON-RPC contract, or the handler contract.
