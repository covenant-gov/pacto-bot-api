Ideation

Decouple Bot SDK and Templates from the pacto-bot-api Release Cycle

Date: 2026-07-02 · Topic: sdk-template-decoupling · Mode: repo-grounded

Grounding Context

pacto-bot-api is a single Rust crate (0.4.1) that contains the daemon, admin CLI (pacto-bot-admin), JSON-RPC contract (schemas/jsonrpc.json), generated Python SDK (python/), and bot project templates (templates/python/). schemas/jsonrpc.json is the canonical source of truth for both Rust generated types (src/*_generated.rs) and the generated Python SDK. cargo xtask codegen regenerates both from the same schema.

Templates are embedded in the admin CLI binary at compile time via include_dir!("templates") and extracted at runtime in src/scaffold/generate.rs. Every template tweak, SDK fix, or scaffold behavior change therefore requires a full daemon/admin CLI release and rebuild. Version numbers are already inconsistent: daemon 0.4.1, Python SDK 0.1.0, schema info.version 0.1.0. There is no independent SDK publishing pipeline or template update mechanism.

External prior art: OpenRPC/Smithy for contract-first versioning; cargo-generate, Cookiecutter, and Copier for language-agnostic templating; MCP's initialize/protocolVersion handshake for JSON-RPC runtime negotiation.

Topic Axes

  1. Contract boundary & versioning — how the JSON-RPC contract version is defined, published, and consumed by SDKs and templates.
  2. SDK release independence — how the Python SDK can be released independently from the daemon and published to PyPI.
  3. Template delivery mechanism — where templates live and how they are fetched (embedded, remote/git-ref, cargo-generate, Cookiecutter/Copier).
  4. Scaffold invocation & tooling — how the admin CLI invokes the templating engine and what engine it uses.
  5. Project drift & update lifecycle — how generated projects know when they are stale and how they update.

Current vs. Decoupled Lifecycle

Combined Vision

The survivor ideas are not mutually exclusive. A coherent end-state is to keep pacto-bot-admin as the single starting point for creating a bot, but turn it into a compatibility-aware bootstrapper that resolves and pulls down three independently released artifacts:

  1. Versioned contract — a published schemas/jsonrpc.json artifact (e.g., pacto-contract@0.1.0).
  2. Versioned SDK — a PyPI package matching the contract version (e.g., pacto-bot-sdk@0.2.0).
  3. Versioned template — a git-tagged cargo-generate template for a given language and bot kind (e.g., pacto-template-python-llm@1.2.0).

The admin CLI consults a compatibility matrix to choose a compatible triple, writes the chosen versions into .pacto/scaffold.lock, and provides an update command to refresh the project when any artifact advances. New languages and bot kinds become new template repositories, not changes to the daemon or CLI.

Why this matters: It preserves the single-entry UX of pacto-bot-admin new while giving every artifact its own release cadence. The contract stays the stable boundary; the CLI becomes the glue that resolves compatibility across the moving parts.

Axis coverage: Contract (#2, #7), SDK (#3), Template (#4, #5), Tooling (#5), Drift (#6, #7).

Ranked Ideas

  1. Cross-cutting: standalone contract + cargo-generate template repo + .pacto/scaffold.lock
  2. Promote schemas/jsonrpc.json to a standalone, versioned contract artifact
  3. Publish Python SDK to PyPI with its own semver and release workflow
  4. Move templates to a separate versioned repository and fetch by git ref
  5. Replace the bespoke template engine with cargo-generate invoked as a subprocess
  6. Write a .pacto/scaffold.lock and add pacto-bot-admin update
  7. Compatibility matrix as a first-class schema consumed by SDKs and templates

1. Cross-cutting: standalone contract + cargo-generate template repo + .pacto/scaffold.lock

Template delivery + Project drift Confidence 85% Complexity High

Description: Combine the three most complementary moves: (1) publish schemas/jsonrpc.json as a standalone versioned artifact, (2) move templates to a cargo-generate repository fetched by git ref, and (3) write a .pacto/scaffold.lock with an update command. The daemon ships the contract, the template repo ships scaffolding, and each project records what it was built from so it can be updated.

Basis: direct: xtask/src/codegen.rs couples Rust and Python generation; src/scaffold/generate.rs:19-20 embeds templates via include_dir!; src/scaffold/generate.rs:600-620 discards manifest.toml and writes no project provenance.

Rationale: No single idea alone solves the full lifecycle. Together they replace the embedded, release-coupled model with a contract-first, template-fetched, lock-tracked model where each piece can move independently.

Downsides: Requires coordinating three architectural changes; introduces a network dependency for fresh templates unless an offline fallback is retained.

2. Promote schemas/jsonrpc.json to a standalone, versioned contract artifact

Contract boundary & versioning Confidence 90% Complexity Medium

Description: Split the contract into its own repository or artifact (e.g., pacto-bot-api-spec) with its own semver, and have the daemon, Python SDK, and templates consume that artifact as a dependency. The schema becomes the source of truth; language generators become consumers.

Basis: direct: xtask/src/codegen.rs:14-26 runs Rust, Python, and compatibility generation from one schemas/ directory; tests/schema_sync.rs:225-266 asserts parity between the schema and the hand-written Method enum, forcing the daemon and schema to move in lockstep. reasoned: versions already diverge (Cargo 0.4.1 vs. schema 0.1.0 vs. Python SDK 0.1.0), proving there is no single source of truth today.

Rationale: Decoupling the spec from the daemon crate is the prerequisite for every other decoupling step. Once the contract is a published artifact, SDKs and templates can target a pinned contract version without waiting for a Rust daemon release.

Downsides: Adds a new repo/artifact to maintain; requires release automation for the contract itself; changes the current one-command cargo xtask codegen workflow.

3. Publish Python SDK to PyPI with its own semver and release workflow

SDK release independence Confidence 85% Complexity Medium

Description: Treat python/ as an independently versioned package. Add a PyPI publishing job to the release workflow, bump the package version on its own schedule, and stop vendoring the SDK into each generated project. Generated bot projects then depend on the published package (pacto-bot-api>=X.Y.Z) instead of a local sdk/ wheel built from a snapshot.

Basis: direct: python/pyproject.toml:7 pins the SDK at 0.1.0; src/scaffold/generate.rs:494-500 copies templates/python/sdk/ into every project; the generated-python-client plan explicitly deferred PyPI publication. external: Hatchling (already in use) supports hatch publish and PyPI upload.

Rationale: Once the SDK ships independently, a bug fix in the Python client no longer requires a daemon release or a rebuilt admin CLI. This is the highest-leverage step for SDK velocity.

Downsides: Requires a separate CI job, PyPI credentials, and a versioning policy for the Python package.

4. Move templates to a separate versioned repository and fetch by git ref

Template delivery mechanism Confidence 85% Complexity Medium

Description: Extract templates/ into a dedicated pacto-bot-templates repository with its own release tags. pacto-bot-admin resolves templates by git ref (defaulting to a pinned tag or branch) instead of embedding them via include_dir!. The admin CLI still ships with a fallback snapshot for offline use, but normal operation fetches the latest compatible template tree.

Basis: direct: src/scaffold/generate.rs:19-20 uses include_dir!("templates") so every template tweak requires a full admin CLI release; templates/python/manifest.toml declares version = "{{version}}" but the renderer ignores it and the manifest is not copied into generated projects. external: cargo-generate and Cookiecutter both support remote git repositories as template sources.

Rationale: Decoupling the template store from the daemon crate means template authors can ship fixes, new languages, and example bots without waiting for a daemon release.

Downsides: Requires network access for fresh templates unless the embedded fallback is kept; adds a git/ref resolution step with error handling.

5. Replace the bespoke template engine with cargo-generate invoked as a subprocess

Scaffold invocation & tooling Confidence 80% Complexity Medium

Description: Replace the custom engine in src/scaffold/template.rs with a std::process::Command invocation of cargo-generate. Keep the existing ScaffoldRequest context as input, but let cargo-generate handle Liquid parsing, conditionals, loops, hooks, and template include paths.

Basis: direct: src/scaffold/template.rs:28-36 implements a bespoke engine supporting only {{key}}, {% if %}, and {% for %}; repo-wide search finds no cargo-generate usage. external: cargo-generate is the de facto Rust template tool and supports cargo-generate.toml, require_version, and Liquid placeholders.

Rationale: Switching to a standard tool replaces a growing maintenance burden with a community-supported engine. It also directly addresses the user's cargo create-style scaffolding suggestion.

Downsides: Adds a runtime dependency on cargo-generate; may require template authors to learn cargo-generate.toml semantics; subprocess error handling is less deterministic than in-process rendering.

6. Write a .pacto/scaffold.lock and add pacto-bot-admin update

Project drift & update lifecycle Confidence 75% Complexity Medium

Description: When scaffolding, write a lock file that records the template version, SDK version, contract version, and admin CLI version that produced the project. Add a pacto-bot-admin update command that reads the lock, fetches the requested template ref, and re-renders files while respecting protected_files and user modifications.

Basis: direct: src/scaffold/generate.rs:260 hard-codes the template version as "0.1.0" and src/scaffold/generate.rs:600-620 skips copying manifest.toml, so generated projects have no recorded provenance; no update|upgrade|migrate command exists in src/. external: cargo-generate supports post-generation hooks and lock-like metadata that can drive updates.

Rationale: Once projects are self-describing, updating a fleet of bots becomes a repeatable command rather than a manual diff. The lock file compounds as a source of truth for drift detection, CI replay, and automated upgrade PRs.

Downsides: Requires an overwrite/merge policy for user-edited files; adds state management and backwards-compatibility concerns for existing projects.

7. Compatibility matrix as a first-class schema consumed by SDKs and templates

Contract boundary & versioning Confidence 75% Complexity Medium

Description: Add a schemas/compatibility.json (or similar) that maps daemon versions to supported JSON-RPC contract versions and compatible SDK/template ranges. The SDK reads it at import or connection time to warn or fail on mismatch; the template engine uses it to select the latest compatible template and SDK defaults; CI uses it to validate release combinations.

Basis: direct: Version numbers are already inconsistent: daemon 0.4.1 (Cargo.toml:3), schema 0.1.0 (schemas/jsonrpc.json:5-6), Python SDK 0.1.0 (python/pyproject.toml:7). schemas/service-compatibility.json already proves the repo can version a separate compatibility surface for backing services. reasoned: without an explicit compatibility model, every release combination is an ad-hoc human judgment.

Rationale: A compatibility matrix makes support and upgrade decisions data-driven and testable. It becomes the authority for the update command, the CI gates, and the SDK handshake.

Downsides: Requires a single owner and update process; if it drifts, it becomes worse than no matrix at all.
